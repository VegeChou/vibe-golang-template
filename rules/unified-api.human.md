  apiDesign:
    pathNaming:
      rules:
        - use lowercase kebab-case for path segments

        - avoid verbs in path; use HTTP methods for action semantics

          - /v1/getUsers
          - /v1/createOrder

      enums: UPPER_SNAKE_CASE
      booleans: prefix with is/has/can when semantically boolean
      datetimeFormat: RFC3339 (ISO-8601) in UTC
      nullability:
        - do not return null for arrays; return []
        - do not return null for objects when empty object semantics are required
    operationConventions:
      - each operation must define summary and description
      - each operation should provide success and error examples
      - use operationId in lowerCamelCase and keep globally unique
    queryConventions:
      filters: use explicit filter keys; avoid free-form query DSL
      sorting:
        fields:
          - sortBy
          - sortOrder
        sortOrderEnum: [asc, desc]

## Exception Handling (from x-rules.exceptionHandling)

```yaml
  exceptionHandling:
    backend:
      - use global exception handler/middleware/filter
      - map exceptions to ApiErrorResponse
      - never expose stack trace to clients in production
      - log traceId, method, path, code, stack
    frontend:
      - handle non-2xx as transport/gateway errors
      - handle 2xx + success=false as business errors
      - redirect on 401
      - show no-permission page on 403
      - include traceId in error reporting
```
